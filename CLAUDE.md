# HDFury Arcana HA Integration

Home Assistant custom integration for the HDFury Arcana eARC adapter,
communicating over RS-232 serial.

## Build Pipeline

**Never invoke pytest, ruff, or pip directly.** Always use `task`.

```
task setup           # create venv, install deps
task test            # full "safe to merge" gate
task format          # auto-format
```

Tasks cache based on source file hashes. `task --force <name>` to bypass.

Requires a nix devshell (`nix develop`) for python, go-task, ruff, and bd.

## Translations

`strings.json` is the source of truth. `translations/en.json` is generated by
`task translations` (wired into the test pipeline). Never edit it directly.

## Serial Protocol Quirks

- Responses are `paramname value\r\n` — the parser strips the param name prefix
- Exception: `ver` returns `ARCANA VER: 0.88\r\n` — special-cased in
  `serial_client.py`
- Some params reject writes depending on physical connection state (earcsel,
  arcsel, audsource, lldvtohdrprimmode)

## Architecture Decisions

- Entities call `coordinator.async_set()`, never `_client` directly
- Entity properties must guard against `coordinator.data` being `None`
- Base entity caches serial/firmware at init (device_info doesn't need live data)
- `async_unload_entry` unloads platforms first, then disconnects serial
- Tests use `pytest-homeassistant-custom-component` — shared fixtures in
  `conftest.py`, per-file coordinator fixtures with test-specific mock data
